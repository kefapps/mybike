# Retrospective courte - Epic 1: Boucle de ride mock jouable

## Ce qui est livre

L'epic MVP livre une vertical slice mock jouable de bout en bout: app shell Vite/React/TypeScript, etats de session, source mock, vitesse lissee, progression, stats, route prefabriquee, camera sur rail, biomes, scene Three.js, HUD minimal, slider, pause/reprise, fin de session et resume reel.

Les cinq stories `MYB-2` a `MYB-6` sont terminees, revuees, committees et synchronisees Linear. Le mode mock reste le mode produit MVP: aucun materiel velo connecte n'est requis.

## Validations finales

- `npm run typecheck` OK.
- `npm run test` OK, validation finale `MYB-6`: 61 tests / 19 fichiers.
- `npm run build` OK, avec warning Vite attendu sur le chunk Three.js > 500 kB.
- Vite HTTP 200 sur `http://127.0.0.1:5174/`.
- Chrome headless/CDP desktop et mobile OK: start, slider mock, pause, reprise, finish, resume reel, canvas crop non blank.
- `test:e2e` non lance car Playwright n'est pas configure; couverture fournie par tests React/Vitest et validation navigateur CDP.

## Limites volontaires / hors scope

- Pas de BLE, Web Bluetooth, FTMS, velo reel, parser Indoor Bike Data ou calibration materiel.
- Pas de backend, compte utilisateur, persistence, cloud sync ou historique riche.
- Pas de Meshy, pipeline assets IA, asset manager lourd ou dependance a des assets externes.
- Pas de routes multiples, editeur de route, import GPX ou selection de route.
- Pas de HUD sportif avance, reglages complets, audio, meteo, cycle jour/nuit ou performance tooling avance.
- Pas de reprise du backlog complet d'archive ni creation d'un large backlog post-MVP.

## Risques restants

- `WebGLRenderer` peut encore echouer apres le preflight WebGL; c'est un hardening preexistant non bloquant, le fallback WebGL indisponible de `MYB-2` reste preserve et teste.
- Le warning de chunk Three.js > 500 kB est acceptable pour le MVP, mais doit etre surveille avant d'ajouter assets et effets.
- La performance est validee qualitativement; aucun budget FPS ou instrumentation avancee n'est encore pose.
- Le happy path navigateur est prouve via CDP et tests React, mais pas automatise avec Playwright.
- Le contenu reste volontairement mince: une route mock, placeholders proceduraux et deux ambiances simples.

## Lecons utiles

- Les frontieres `ride` pur, `route` pur et `render` consommateur de snapshots ont bien contenu le scope et les revues.
- Les corrections les plus utiles sont apparues aux interfaces: routes malformees, config camera `null`, effort mock controle et horloge de pause.
- Garder le mock comme experience produit, pas seulement comme outil de debug, a rendu la vertical slice verifiable sans dependances externes.

## Options post-MVP priorisees

1. Hardening technique leger: securiser l'echec `WebGLRenderer` apres preflight et automatiser un happy path navigateur minimal si Playwright devient pertinent.
2. Playtest UX mock: confort camera, lisibilite HUD, ressenti du slider, comprehension du resume.
3. Polissage visuel contenu: enrichir placeholders, transitions de biomes et lisibilite de route sans introduire de pipeline lourd.
4. Preparation telemetry: cadrer une source `bleFtms` derriere `RideInputSource`, sans changer la boucle mock ni l'activer par defaut.
5. Strategie performance/assets: poser un budget simple avant d'ajouter assets externes, effets ou routes supplementaires.

## Cloture

Verdict: Epic 1 clos cote MVP mock. Prochaine decision recommandee: choisir une seule option post-MVP avant de creer de nouvelles issues.
